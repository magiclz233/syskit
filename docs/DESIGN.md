# 设计说明

## 设计目标

这个项目只解决一件事：准确找出目标目录树中最大的子目录和最大的文件。

因此设计上做了两个明确取舍：

- 不再提供近似扫描模式
- 不用“只扫前几层”或“先猜大目录再补扫”的策略

## 扫描策略

当前实现使用单一准确扫描器，基于 `filepath.WalkDir` 顺序遍历整个目录树。

遍历过程中做三件事：

1. 记录文件总数、目录总数和累计字节数
2. 把每个文件的大小累加到其所有父目录
3. 维护 Top N 文件候选集

最终输出时：

- 子目录列表按累计大小排序
- 文件列表按单文件大小排序
- 根目录本身从子目录结果中排除

## 为什么只保留单一扫描器

这个需求要求结果完整、可解释，所以实现上只保留一套准确扫描逻辑。

这样做的好处是：

- 不会因为扫描范围裁剪而漏掉深层大文件
- 不会因为启发式目录筛选而漏掉真正的大文件
- CLI、脚本、文档和输出语义可以长期保持一致

## 结果完整性

为了保证输出可解释，当前版本遵循下面的规则：

- 子目录大小是累计值
- 文件排名基于实际字节大小
- 无权限路径跳过，但不会终止扫描
- 符号链接跳过，避免循环

## 内存策略

目录大小必须为每个扫描到的目录保留一条聚合记录，这是准确计算 Top 子目录所必需的。

文件结果不需要保留全部文件，因此实现中会周期性裁剪文件候选集，只保留足够多的 Top 候选，避免在超大目录树下无上限增长。
