## 2. 产品定位

`syskit` 是一款面向开发者、中小团队运维工程师和测试工程师的跨平台本地系统运维 CLI 工具。

核心能力闭环：

1. 检查：快速读取本机状态。
2. 诊断：基于规则输出问题、证据、影响和建议。
3. 修复：在可控风险前提下执行自动修复。
4. 监控：持续观察并告警。
5. 巡检：定时执行检查与诊断，生成报告。
6. 复盘：通过快照和报告追踪变化。

## 3. 产品目标

1. 将本地系统故障平均定位时间缩短 50% 以上。
2. 提供“一条命令定位问题，一条命令执行修复”的高频闭环。
3. 在 Windows、Linux、macOS 上提供稳定、一致、可脚本化的 CLI 体验。
4. 默认安全，所有写操作可审计、可确认、可回看。
5. 在 CI/CD、手工排障、日常巡检三类场景下复用同一套输出协议。

## 4. 非目标

1. 不做远程主机管理平台。
2. 不做重型分布式监控平台替代品。
3. 不做内核级 APM 和分布式追踪。
4. P0-P1 不做 GUI 或 Web 控制台。
5. P0-P1 不做多主机批量运维和集群编排。

## 5. 用户与核心场景

### 5.1 目标用户

| 角色 | 核心诉求 | 使用频率 |
|---|---|---|
| 开发者 | 端口冲突、服务启动失败、机器卡顿、磁盘爆满 | 高频 |
| 测试工程师 | 环境巡检、快照留存、自动清理、复盘 | 中频 |
| 运维/平台工程师 | 健康体检、策略校验、报告输出、定时巡检 | 高频 |
| 个人站长/技术爱好者 | 资源观察、异常排查、日常维护 | 中频 |

### 5.2 核心场景

1. 开发环境故障排查：端口占用、残留进程、环境异常。
2. 系统性能优化：CPU、内存、磁盘、网络异常定位。
3. 日常巡检：一键体检、策略检查、风险汇总。
4. 磁盘治理：大文件、重复文件、垃圾文件、日志归档。
5. 故障复盘：快照创建、快照对比、报告导出。
6. 持续监控：阈值告警、日志增长、资源趋势分析。

## 6. 产品原则

1. KISS：命令默认开箱即用，必要参数最少。
2. YAGNI：P0 只交付最高频闭环，不把所有能力挤进 MVP。
3. DRY：命令、输出、规则、架构各自只有一个真相源。
4. SOLID：CLI 协议、规则引擎、平台适配、存储、输出渲染相互解耦。
5. Safe by default：所有写操作默认 dry-run，显式 `--apply` 才执行。

## 7. 产品能力地图

### 7.1 能力总览

| 模块 | 核心价值 | 代表命令 | 版本 |
|---|---|---|---|
| Doctor | 一键体检和专项诊断 | `doctor all` | P0 |
| Port | 端口查询、释放、连通性、扫描 | `port`、`port kill` | P0/P1 |
| Proc | 进程排行、树、详情、结束 | `proc top`、`proc kill` | P0 |
| CPU | CPU 总览、突发捕捉、持续监控 | `cpu`、`cpu burst` | P0/P1 |
| Mem | 内存总览、排行、泄漏趋势、监控 | `mem`、`mem leak` | P0/P1 |
| Disk/File | 分区总览、扫描、重复文件、清理、归档 | `disk`、`disk scan` | P0/P1 |
| Net | 连接审计、DNS、Ping、Traceroute、测速 | `net conn`、`doctor network` | P1 |
| Service | 服务列表、健康检查、启停、自启管理 | `service list` | P1 |
| Startup | 启动项扫描与管理 | `startup list` | P1 |
| Log | 日志体检、搜索、增长监控 | `log`、`log search` | P1 |
| Fix | 单动作修复与剧本编排 | `fix cleanup`、`fix run` | P0/P1 |
| Monitor | 全系统监控、单维度监控、定时巡检 | `monitor all` | P1 |
| Snapshot/Report | 快照创建、对比、报告导出 | `snapshot create`、`report` | P0/P1 |
| Policy | 配置与策略查看、初始化、校验 | `policy show`、`policy validate` | P0 |

### 7.2 模块范围说明

#### P0 核心闭环

1. `doctor all` 和核心专项诊断：`doctor port/cpu/mem/disk`。
2. 高频操作：`port`、`port list`、`port kill`、`proc top`、`proc tree`、`proc info`、`proc kill`。
3. 资源总览：`cpu`、`mem`、`disk`。
4. 核心磁盘治理：`disk scan`、`fix cleanup`。
5. 快照与报告基础能力：`snapshot create/list/diff/show/delete`、`report`。
6. 配置与策略基础能力：`policy show/init/validate`。
7. 三平台适配、统一输出、错误码、审计、权限降级。

#### P1 扩展能力

1. 网络工具集：`port ping`、`port scan`、`net conn/listen/speed`、`dns resolve/bench`、`ping`、`traceroute`、`doctor network`。
2. 深度监控：`cpu watch`、`mem watch`、`mem leak`、`monitor all`。
3. 运维对象：`service *`、`startup *`、`log *`。
4. 文件治理增强：`file dup`、`file dedup`、`file archive`、`file empty`。
5. 修复编排：`fix run` 和自定义 YAML 剧本。
6. 定时巡检、Webhook 告警、模板化报告。

#### P2 高级能力

1. 插件机制与自定义规则扩展。
2. 容器/K8s 诊断。
3. Web 可视化报告和更强交互界面。
4. 多主机批量巡检能力。

## 9. 功能需求摘要

### 9.1 Doctor

1. `doctor all` 输出健康分、健康等级、覆盖率、问题清单、跳过模块。
2. 每条问题必须包含规则 ID、严重级别、摘要、证据、影响、建议、修复命令、是否可自动修复、置信度、作用域。
3. 支持 `doctor port/cpu/mem/disk/network/disk-full/slowness` 等专项诊断。
4. 单模块失败不应拖垮全量诊断，必须记录 `skipped`。

### 9.2 高频运维命令

1. 端口：查询、列表、释放、连通性、扫描。
2. 进程：TopN、树、详情、结束。
3. 资源：CPU、内存、磁盘总览和持续观察。
4. 文件：大文件、大目录、重复文件、空目录、旧日志治理。

### 9.3 运维对象命令

1. 服务：列表、健康检查、启停重启、自启管理。
2. 启动项：清单、风险标记、启用禁用。
3. 日志：体检、搜索、增长监控。

### 9.4 修复与编排

1. 所有写操作默认 dry-run。
2. 真实执行必须 `--apply`。
3. dangerous 级别操作必须 `--apply --yes`。
4. 剧本支持串行、并行、条件、失败策略、平台条件。

### 9.5 监控、巡检、复盘

1. 监控支持阈值告警、终端提醒、本地日志、Webhook。
2. 定时巡检本质上是“按计划执行 `doctor all` 并保存报告”。
3. 快照要能创建、列出、查看、删除和对比。
4. 报告要支持 health、inspection、monitor 三种类型。

### 9.6 配置与策略

1. 配置文件负责默认参数、阈值、存储、隐私和风险控制。
2. 策略文件负责 required rules、阈值覆盖、关键服务、禁止进程等团队标准。
3. `policy` 命令同时支持查看、初始化、校验配置文件和策略文件。

## 10. 统一业务约束

### 10.1 安全与权限

1. 只读命令默认普通用户可运行。
2. 系统级写操作要求管理员/root。
3. 权限不足时降级执行，必须输出覆盖率和跳过项，不允许静默吞错。
4. 所有写操作写入审计日志。
5. 系统关键进程、核心服务默认白名单保护。

### 10.2 输出与协议

1. 所有命令必须支持结构化输出。
2. 统一输出格式为 `table/json/markdown/csv`，具体命令按场景限制默认值。
3. JSON 字段采用 snake_case。
4. JSON 协议遵循“只增不减”。

### 10.3 配置与存储

1. 配置优先级：命令行 > 用户配置 > 系统配置 > 默认值。
2. 默认保留策略：`retention_days=14`，`max_storage_mb=500`。
3. 存储内容：快照、监控数据、报告、审计日志。
4. 存储清理要可追踪并加锁，避免并发破坏。

### 10.4 健康分与结果判定

1. 健康分基础分为 100。
2. 严重级别、置信度、作用域共同影响扣分。
3. `coverage` 单独表示本次检查覆盖程度，不与 `health_score` 混为一谈。
4. `--fail-on` 控制 CI/CD 阻断阈值。

## 11. 非功能需求

1. `doctor all --mode quick` 常规环境 30 秒内完成，深度模式 3 分钟内完成。
2. 查询类命令常规场景 2 秒内返回。
3. 大目录扫描性能不低于当前仓库迁移前的大文件扫描能力基线。
4. 大规模目录扫描和重复文件检测不得因全量载入导致内存失控。
5. 单模块失败、权限不足、平台不支持三类场景都要可降级。
6. 所有命令在 Windows、Linux、macOS 上保持相同的字段语义。

## 12. 验收标准

### 12.1 功能验收

1. P0 交付范围内命令全部可运行。
2. 端口冲突排查到修复不超过两条命令。
3. `doctor all` 必须输出完整问题清单与健康分。
4. 快照创建和快照对比结果可追溯。
5. 配置与策略文件可以初始化、查看、校验。

### 12.2 安全验收

1. 无 `--apply` 时不发生任何真实写操作。
2. dangerous 操作无 `--yes` 不可执行。
3. 审计日志覆盖所有写操作。
4. 权限不足时输出清晰，不泄露不该展示的信息。

### 12.3 协议验收

1. JSON 输出字段与 CLI 规范一致。
2. 规则字段与规则目录一致。
3. 新版本只能新增可选字段，不能删除既有字段。
4. 退出码和 `--fail-on` 行为可用于脚本和 CI。

## 13. 开发完成定义（DoD）

1. 命令帮助、示例、错误提示完整。
2. 核心命令具备单元测试、集成测试、契约测试。
3. 权限不足、超时、平台不支持三类异常都正确降级。
4. 三平台编译通过，P0 命令在三平台完成验证。
5. 文档同步更新，不出现 PRD、CLI、架构、规则目录互相打架的情况。

## 14. 后续文档协同要求

1. 新增命令或参数时，先更新 `SYSKIT_CLI_SPEC.md`。
2. 新增输出字段或内部模型时，先更新 `SYSKIT_ARCHITECTURE.md`。
3. 新增或修改规则时，先更新 `SYSKIT_RULES_CATALOG.md`。
4. 只有影响范围、优先级、验收或用户价值时，才更新本 PRD。
