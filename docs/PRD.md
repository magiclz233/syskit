# syskit 产品需求文档（PRD）v1.3 最终版
**文档状态**：最终定稿 | **生效日期**：2026-03-12 | **产品定位**：跨平台一站式本地系统运维CLI工具集

---

## 1. 产品概述
### 1.1 产品定位
syskit 是一款**面向开发者、中小团队运维工程师**的跨平台本地系统运维CLI工具，核心提供「**检查（Inspect）-诊断（Doctor）-修复（Fix）-监控（Monitor）-巡检（Inspection）-报告（Report）**」全闭环能力，解决本地系统运维的高频痛点。

核心价值：比系统原生命令更快定位问题、比重型监控工具更轻量易部署、支持脚本化集成、可控执行风险、1条命令完成全链路故障处理。

### 1.2 产品目标
1.  核心效率：将本地系统故障平均定位时间缩短50%以上，常见问题实现「1条命令定位+1条命令修复」闭环；
2.  兼容性：100%兼容 Windows、Linux、macOS 三大主流操作系统，同一条命令在三平台输出格式、功能逻辑完全一致；
3.  易用性：默认只读操作，新手零风险上手；高频命令提供短别名，老用户高效使用；所有命令支持结构化输出，无缝对接脚本/CI/CD；
4.  扩展性：模块化架构，新增功能无需修改核心代码，可快速迭代扩展。

### 1.3 非目标（边界定义）
1.  不做远程主机管理平台，仅聚焦本地单主机运维能力；
2.  不做 Prometheus/Grafana 等重型分布式监控系统的替代品，仅提供本地实时监控与阈值告警；
3.  不做内核级APM全链路追踪能力，仅提供用户态系统级、进程级资源诊断；
4.  P0-P1版本不做GUI桌面端、Web可视化平台，仅聚焦CLI核心能力；
5.  不做多主机批量运维、集群管理能力，仅聚焦本地单主机场景。

## 2. 用户画像与核心场景
### 2.1 用户画像
| 用户角色 | 核心诉求 | 使用频率 |
|----------|----------|----------|
| 后端/前端开发者 | 快速解决端口占用、进程残留、服务启动失败、机器卡顿、磁盘爆满等开发环境问题 | 高频（每日多次） |
| 测试工程师 | 测试环境前置巡检、服务健康检查、故障快照留存、测试环境清理 | 中频（每日1-2次） |
| 中小团队运维工程师 | 服务器日常巡检、故障快速定位、常规运维操作自动化、巡检报告生成 | 高频（每日多次） |
| 个人站长/技术爱好者 | 个人服务器日常维护、资源监控、风险排查 | 中频（每周多次） |

### 2.2 核心用户故事
1.  作为后端开发者，我想要1条命令查到8080端口被哪个进程占用，并一键释放端口，以便快速启动我的后端服务；
2.  作为开发者，我发现电脑突然变卡，想要1条命令快速定位是哪个进程占用了过高的CPU/内存，以便结束异常进程；
3.  作为运维工程师，我想要1条命令完成服务器全量体检，输出健康评分和所有风险点，并给出可执行的修复命令，以便完成日常巡检；
4.  作为开发者，我想要1条命令完成磁盘空间分析，找到大文件、重复文件、临时垃圾文件，并一键清理，以便解决磁盘爆满问题；
5.  作为运维工程师，我想要配置定时巡检任务，每天自动生成服务器健康报告，出现严重风险时触发告警，以便提前规避故障。

### 2.3 核心使用场景
1.  开发环境故障排查：端口冲突、服务启动失败、进程残留、环境异常；
2.  系统性能优化：CPU/内存/磁盘IO异常定位、资源占用优化、泄漏检测；
3.  日常运维巡检：服务器健康体检、风险点排查、合规性检查；
4.  磁盘空间治理：大文件/重复文件/垃圾文件清理、日志归档；
5.  网络故障诊断：端口不通、DNS解析异常、网络链路故障、异常连接排查；
6.  故障复盘：系统状态快照留存、故障前后状态对比、根因定位。

## 3. 核心架构设计
### 3.1 整体分层架构
采用「**CLI入口层-分析层-执行层-采集层-平台适配层**」五层解耦架构，严格遵循SOLID/DRY/KISS原则，所有模块可复用、可扩展、可替换。

| 架构层 | 核心职责 | 包含模块 |
|--------|----------|----------|
| CLI入口层 | 命令解析、参数校验、子命令调度、全局参数处理、用户交互 | 基于cobra的命令框架、所有子命令入口、全局参数管理 |
| 分析层 | 采集数据加工、规则匹配、根因分析、健康分计算、问题分级 | 诊断规则引擎、健康分计算器、数据聚合分析、阈值判断 |
| 执行层 | 修复动作执行、剧本编排、安全校验、权限检查、操作审计、执行结果反馈 | 修复执行器、剧本编排引擎、安全校验器、审计日志模块 |
| 采集层 | 系统/进程/网络/磁盘等原始数据采集、数据标准化、过滤处理 | 各功能模块采集器、数据标准化转换器、过滤规则处理器 |
| 平台适配层 | 屏蔽三平台系统差异，提供统一的采集接口，处理平台兼容性问题 | Windows/Linux/macOS 分平台适配实现、统一接口封装 |

**目录设计原则**：
1. `cmd/<binary>` 只承载可执行入口，便于后续扩展多二进制
2. `internal/cli` 按模块分目录，避免单文件过大，每个子命令组独立维护
3. `internal/app` 作为应用服务层，负责业务编排和模块调度
4. `internal/domain` 作为领域层，包含规则引擎、评分器等核心逻辑
5. `internal/collectors` 统一命名为复数形式，与 ARCHITECTURE.md 保持一致
6. `platform` 独立出来，不放在 internal 下，便于跨平台能力复用
7. `pkg` 只放可对外暴露的公共包，内部包统一放 internal

## 4. 全局通用规范（所有命令必须遵守）
### 4.1 全局通用参数
所有子命令均支持以下全局参数，无需重复定义：
| 参数全称 | 短名 | 类型 | 默认值 | 功能描述 |
|----------|------|------|--------|----------|
| --help | -h | bool | false | 查看当前命令帮助文档 |
| --version | -v | bool | false | 查看syskit版本号 |
| --format | -f | string | table | 输出格式，可选值：table/json/csv |
| --json | 无 | bool | false | 快捷设置输出格式为json，优先级高于--format |
| --quiet | -q | bool | false | 静默模式，仅输出错误信息和核心结果 |
| --verbose | -V | bool | false | 详细模式，输出全量调试信息 |
| --dry-run | 无 | bool | true | 试运行模式，所有写操作仅模拟执行，不修改系统（仅fix类命令生效） |
| --apply | 无 | bool | false | 确认执行写操作，必须显式指定，关闭dry-run模式 |
| --yes | -y | bool | false | 跳过危险操作的二次确认，仅与--apply同时生效 |

### 4.2 输出格式规范
1.  **终端表格输出**：人类友好格式，列宽自适应，关键信息高亮，异常数据标红；
2.  **JSON输出**：机器友好格式，所有命令输出结构统一，包含`code`（退出码）、`msg`（提示信息）、`data`（核心数据）、`error`（错误信息）四个根字段；
3.  **CSV输出**：可直接导入Excel，表头与表格输出列名完全一致，支持--export-csv参数导出为文件。

### 4.3 退出码规范
所有命令执行完成后返回统一的退出码，便于脚本/CI/CD集成判断执行结果：
| 退出码 | 含义 | 适用场景 |
|--------|------|----------|
| 0 | 执行成功，无异常/警告 | 命令执行完成，无任何错误、无风险点 |
| 1 | 执行成功，存在警告级问题 | 命令执行完成，仅存在medium/low级风险，无严重错误 |
| 2 | 执行失败，存在严重级问题 | 命令执行完成，存在critical/high级风险，或CI/CD阻断场景 |
| 3 | 参数错误 | 命令行参数输入错误、必填参数缺失、参数值不合法 |
| 4 | 权限不足 | 当前用户无执行该命令的权限，或部分数据采集权限不足 |
| 5 | 执行失败 | 命令执行过程中出现不可恢复的错误，无法完成核心功能 |
| 6 | 部分执行成功 | 批量操作中部分任务成功，部分失败（如批量端口扫描） |

### 4.4 安全等级规范
所有写操作（修改系统配置、结束进程、删除文件等）按风险等级分为三级，执行规则严格区分：
| 安全等级 | 定义 | 执行要求 | 示例场景 |
|----------|------|----------|----------|
| safe | 安全操作，无任何系统破坏性，仅修改工具自身数据 | 无需--apply，可直接执行 | 生成快照、导出报告、创建配置文件 |
| cautious | 谨慎操作，可能影响正在运行的业务，可回滚 | 必须显式指定--apply才能执行，默认dry-run | 结束用户进程、重启用户服务、清理临时文件 |
| dangerous | 危险操作，可能影响系统稳定性，不可逆 | 必须同时指定--apply和--yes才能执行，强制二次确认 | 结束系统进程、禁用系统启动项、删除系统文件、修改系统配置 |

### 4.5 配置文件规范
1.  **配置文件优先级**：命令行参数 > 用户级配置文件 > 系统级配置文件 > 工具默认配置
2.  **配置文件路径**：
    | 平台 | 系统级配置路径 | 用户级配置路径 |
    |------|----------------|----------------|
    | Linux | /etc/syskit/config.yaml | ~/.syskit/config.yaml |
    | Windows | %ProgramData%/syskit/config.yaml | %USERPROFILE%/.syskit/config.yaml |
    | macOS | /Library/Application Support/syskit/config.yaml | ~/.syskit/config.yaml |
3.  **配置文件格式**：YAML格式，支持阈值规则、默认参数、排除规则、告警配置等。

## 5. 分模块详细功能需求（细化到最小功能点）
### 5.1 核心入口模块：doctor（一键体检+专项诊断）
**模块定位**：产品核心杀手级功能，用户无需知道具体命令，1条命令完成全系统扫描，输出健康评分、问题清单、修复方案。

#### 5.1.1 全局命令：syskit doctor all（一键全量体检）
| 项目 | 详细规范 |
|------|----------|
| 命令别名 | syskit checkup、syskit health |
| 功能描述 | 一键扫描全系统所有风险维度，输出健康评分、按严重等级排序的问题清单、每条问题的修复建议和可执行命令 |
| 权限要求 | 普通用户可执行（仅扫描当前用户有权限访问的内容），root/管理员权限可执行全量扫描 |
| 跨平台兼容性 | 100%兼容Windows/Linux/macOS，扫描维度自动适配平台 |
| 核心参数 | 见下表 |
| 执行耗时 | 快速模式≤30s，深度模式≤3min |

##### 核心参数明细
| 参数全称 | 短名 | 类型 | 默认值 | 功能描述 |
|----------|------|------|--------|----------|
| --mode | -m | string | quick | 扫描模式，可选值：quick（快速扫描，仅高频风险项）、deep（深度扫描，全量风险项） |
| --output | -o | string | 终端输出 | 报告导出路径，支持.md/.json/.csv格式，如--output report.md |
| --fail-on | string | 无 | 失败阈值，可选值：critical/high/medium/low，对应等级问题存在时返回退出码2，用于CI/CD阻断 |
| --apply-safe | 无 | bool | false | 扫描完成后自动执行所有safe级可自动修复的问题 |
| --exclude | -e | string | 无 | 排除扫描的模块，多个用逗号分隔，如--exclude log,startup |

##### 最小功能点明细
1.  自动加载配置文件中的扫描规则和阈值，支持用户自定义扫描项；
2.  按模块并行扫描，单个模块扫描失败不影响其他模块执行，最终汇总失败模块清单；
3.  扫描维度覆盖：端口、进程、CPU、内存、磁盘、网络、服务、启动项、日志9大核心模块；
4.  对扫描结果进行规则匹配，识别风险点，按统一的问题结构生成问题清单；
5.  按问题严重等级（critical>high>medium>low）对问题清单排序，优先展示最高风险问题；
6.  基于问题清单计算系统健康评分，输出评分和风险等级；
7.  终端输出包含：健康评分、风险等级概览、问题清单（每条含等级、描述、证据、影响、修复命令）、失败模块汇总；
8.  支持导出Markdown/JSON/CSV格式的完整体检报告；
9.  支持自动执行安全级修复动作，执行完成后输出修复结果；
10. 扫描完成后自动生成系统快照，留存当前系统状态，用于后续对比。

##### 健康分计算规则
1.  基础分值：100分
2.  扣分规则：
    | 问题严重等级 | 单条问题扣分 | 单等级最大扣分 |
    |--------------|--------------|----------------|
    | critical（严重） | 30分/条 | 100分 |
    | high（高危） | 15分/条 | 80分 |
    | medium（中危） | 5分/条 | 30分 |
    | low（低危） | 1分/条 | 10分 |
3.  最终分值：`max(0, 100 - 总扣分)`
4.  健康等级划分：
    | 分数区间 | 健康等级 | 终端标识 |
    |----------|----------|----------|
    | 90-100 | 优秀 | 绿色 |
    | 70-89 | 良好 | 蓝色 |
    | 60-69 | 警告 | 黄色 |
    | 0-59 | 高危 | 红色 |

##### 统一问题结构（所有问题必须遵循）
| 字段名 | 类型 | 取值范围 | 字段含义 |
|--------|------|----------|----------|
| id | string | 模块-编号，如PORT-001 | 问题唯一标识 |
| severity | string | critical/high/medium/low | 问题严重等级 |
| category | string | port/proc/cpu/mem/disk/net/service/startup/log | 问题所属模块 |
| summary | string | 无 | 一句话问题描述 |
| evidence | string | 无 | 问题关键证据，如端口号、PID、路径、指标值 |
| impact | string | 无 | 问题可能造成的影响 |
| suggestion | string | 无 | 人工修复建议 |
| fix_command | string | 无 | 可直接执行的自动修复命令 |
| auto_fixable | bool | true/false | 是否支持自动修复 |
| fix_safety_level | string | safe/cautious/dangerous | 修复动作的安全等级 |
| confidence | int | 0-100 | 问题判断的置信度，100为完全确认 |

##### P0版本内置扫描规则（10条核心）
| 规则ID | 严重等级 | 规则描述 | 判断逻辑 |
|--------|----------|----------|----------|
| PORT-001 | critical | 端口冲突 | 常用端口（80/443/8080/3306/6379等）被非预期进程占用，或同端口多进程监听冲突 |
| DISK-001 | critical | 磁盘分区空间不足 | 磁盘分区使用率≥90%，或剩余空间≤5GB |
| PROC-001 | high | 进程CPU占用过高 | 单个进程CPU使用率≥90%，持续时间≥30s |
| PROC-002 | high | 进程内存占用过高 | 单个进程内存占用≥系统总内存的50% |
| MEM-001 | high | 系统内存压力过大 | 系统物理内存使用率≥90%，且Swap分区使用率≥50% |
| CPU-001 | high | 系统CPU负载过高 | 系统1分钟平均负载≥CPU核心数*2，持续时间≥1min |
| SERVICE-001 | high | 关键服务异常 | 用户配置的关键服务未运行，或开机自启但启动失败 |
| DISK-002 | high | 磁盘增长过快 | 磁盘分区日增长≥10GB，预计7天内会写满 |
| NET-001 | medium | 异常网络外连 | 进程建立了非公网IP的外连，或连接数≥100的异常长连接 |
| STARTUP-001 | medium | 可疑开机启动项 | 启动项无签名、路径为临时目录、无已知发布者信息 |

#### 5.1.2 专项诊断命令
| 命令 | 功能描述 | 核心参数 |
|------|----------|----------|
| syskit doctor port <端口号> | 端口冲突专项诊断，输出端口占用全链路：端口→PID→进程名→启动命令→父进程→是否可安全释放 | --detail（输出详细进程信息） |
| syskit doctor disk-full | 磁盘爆满专项诊断，输出大文件/大目录/重复文件/日志文件/临时文件清单，以及可回收空间统计 | --path <指定路径>、--top <数量> |
| syskit doctor slowness | 系统卡顿专项诊断，联合分析CPU/内存/磁盘IO/进程/启动项，输出卡顿根因排序和优化建议 | --mode quick/deep |
| syskit doctor cpu | CPU异常专项诊断，输出高CPU进程、系统负载情况、CPU瓶颈根因 | --burst（捕捉突发高CPU进程） |
| syskit doctor mem | 内存异常专项诊断，输出系统内存压力、高内存进程、内存泄漏风险进程 | --detail（输出内存细分指标） |
| syskit doctor network | 网络异常专项诊断，一键完成DNS→网关→公网全链路检测，输出网络故障根因 | --target <目标地址> |

### 5.2 端口管理模块：port
**模块定位**：最高频核心功能，解决端口占用、端口可达性、端口扫描等核心痛点，完全覆盖用户需求。

#### 5.2.1 端口占用查询：syskit port <端口号>
| 项目 | 详细规范 |
|------|----------|
| 命令别名 | syskit port check、syskit inspect port |
| 功能描述 | 查询指定端口的占用情况，反向关联占用进程的全量信息 |
| 权限要求 | 普通用户可查询当前用户进程的端口占用，root/管理员可查询全量端口 |
| 跨平台兼容性 | 100%兼容Windows/Linux/macOS |
| 核心参数 | --detail（输出详细进程信息）、--json（结构化输出） |

##### 最小功能点明细
1.  支持单端口查询，如`syskit port 8080`；
2.  支持多端口批量查询，多个端口用逗号分隔，如`syskit port 8080,3306,6379`；
3.  支持端口范围查询，如`syskit port 8000-9000`；
4.  输出字段包含：端口号、协议类型（TCP/UDP）、监听地址、进程PID、进程名、完整启动命令、运行用户、进程启动时间、父进程PID；
5.  自动识别进程类型，标注是否为开发工具进程（node/java/python/dotnet等）、系统进程、业务服务进程；
6.  自动判断是否可安全结束进程，给出风险提示；
7.  端口未被占用时，给出明确提示，无错误输出；
8.  无权限查询的端口，标注权限不足，不中断整体查询。

#### 5.2.2 监听端口列表：syskit port list
| 项目 | 详细规范 |
|------|----------|
| 命令别名 | syskit port ls、syskit net listen |
| 功能描述 | 列出系统所有正在监听的端口，按进程聚合展示 |
| 权限要求 | 普通用户可查询当前用户进程的监听端口，root/管理员可查询全量 |
| 跨平台兼容性 | 100%兼容三平台 |
| 核心参数 | --by <pid/port>（排序维度）、--protocol <tcp/udp>（协议过滤）、--listen <0.0.0.0/127.0.0.1>（监听地址过滤） |

##### 最小功能点明细
1.  输出全量监听端口，默认按端口号升序排序；
2.  支持按PID、进程名、协议、监听地址过滤；
3.  输出字段包含：端口号、协议、监听地址、PID、进程名、运行用户、启动时间；
4.  自动标注风险端口：0.0.0.0监听的公网暴露端口、高危端口（22/3389等）；
5.  支持按进程聚合，展示单个进程监听的所有端口。

#### 5.2.3 释放端口：syskit port kill <端口号>
| 项目 | 详细规范 |
|------|----------|
| 命令别名 | syskit fix port、syskit kill-port |
| 功能描述 | 结束占用指定端口的进程，释放端口 |
| 安全等级 | cautious（用户进程）/dangerous（系统进程） |
| 权限要求 | 普通用户可结束当前用户的进程，root/管理员可结束所有进程 |
| 跨平台兼容性 | 100%兼容三平台 |
| 核心参数 | --force（强制结束进程）、--kill-tree（结束进程及其所有子进程）、--apply（确认执行）、--yes（跳过二次确认） |

##### 最小功能点明细
1.  默认dry-run模式，仅输出将要结束的进程信息，不实际执行；
2.  必须显式指定--apply才会实际执行结束进程操作；
3.  支持单端口、多端口批量释放；
4.  结束进程前，先校验端口占用情况，确认进程存在；
5.  系统进程、关键服务进程默认禁止结束，除非同时指定--force和--yes；
6.  支持结束进程树，同时结束占用端口进程的所有子进程；
7.  执行完成后，再次校验端口占用情况，输出释放结果（成功/失败）；
8.  所有操作写入审计日志，记录执行时间、用户、端口、PID、执行结果。

#### 5.2.4 端口可达性测试：syskit port ping <目标地址> <端口号>
| 项目 | 详细规范 |
|------|----------|
| 功能描述 | 测试目标地址的指定TCP端口是否可连通，输出连通性、延迟、超时情况 |
| 跨平台兼容性 | 100%兼容三平台 |
| 核心参数 | --count <次数>（默认4次）、--timeout <超时时间>（默认3s）、--interval <间隔>（默认1s） |

##### 最小功能点明细
1.  支持单个地址+端口测试，如`syskit port ping example.com 80`；
2.  支持多地址+多端口批量测试，如`syskit port ping example.com 80,443`；
3.  输出字段：目标地址、端口、连通状态、平均延迟、丢包率、超时次数；
4.  端口不通时，自动诊断根因：是目标地址不可达、端口未监听、防火墙拦截、超时；
5.  支持连续测试模式，--count 0 持续测试，直到用户手动终止。

#### 5.2.5 端口扫描：syskit port scan <目标地址>
| 项目 | 详细规范 |
|------|----------|
| 功能描述 | 扫描目标地址的端口开放情况，识别开放端口、服务类型 |
| 跨平台兼容性 | 100%兼容三平台 |
| 核心参数 | --port <端口范围>（默认1-1000）、--mode <quick/full>（默认quick）、--timeout <超时时间>（默认500ms） |

##### 最小功能点明细
1.  支持快速扫描（常用1000个端口）和全端口扫描（1-65535）；
2.  支持指定端口范围扫描，如--port 80-1000；
3.  支持指定端口列表扫描，如--port 80,443,3306,6379；
4.  输出字段：端口号、协议、开放状态、服务类型（HTTP/HTTPS/MySQL/Redis等）、响应延迟；
5.  扫描完成后，汇总开放端口清单，按端口号排序；
6.  支持并发扫描，可自定义并发数，默认并发100，避免扫描超时。

### 5.3 进程管理模块：proc
**模块定位**：进程资源诊断、进程管理、异常进程定位，完全覆盖用户需求。

#### 5.3.1 进程资源排行：syskit proc top
| 项目 | 详细规范 |
|------|----------|
| 命令别名 | syskit top（高频核心别名） |
| 功能描述 | 按CPU/内存/磁盘IO等维度排序进程，输出TopN进程资源占用情况 |
| 权限要求 | 普通用户可查看当前用户进程，root/管理员可查看全量进程 |
| 跨平台兼容性 | 100%兼容三平台 |
| 核心参数 | 见下表 |

##### 核心参数明细
| 参数全称 | 短名 | 类型 | 默认值 | 功能描述 |
|----------|------|------|--------|----------|

```yaml
    description: "要释放的端口号"
  - name: service_name
    type: string
    required: true
    description: "要重启的服务名"

# 执行步骤
steps:
  - name: "check-port-occupation"
    description: "检查端口占用情况"
    action: "port.check"
    params:
      port: "{{ .port }}"
    # 条件判断，端口被占用才继续执行
    when: "result.occupied == true"
    # 失败策略
    on_fail: "stop"

  - name: "kill-port-process"
    description: "结束占用端口的进程"
    action: "port.kill"
    params:
      port: "{{ .port }}"
    on_fail: "stop"
    # 平台适配，仅在Linux/macOS执行
    platforms: ["linux", "darwin"]

  - name: "stop-service"
    description: "停止目标服务"
    action: "service.stop"
    params:
      service_name: "{{ .service_name }}"
    on_fail: "continue"

  - name: "start-service"
    description: "启动目标服务"
    action: "service.start"
    params:
      service_name: "{{ .service_name }}"
    on_fail: "stop"

  - name: "health-check"
    description: "服务健康检查"
    action: "service.check"
    params:
      service_name: "{{ .service_name }}"
    # 重试配置
    retry:
      max: 3
      interval: 2s
    on_fail: "stop"
```

## 6. 权限模型与安全机制
### 6.1 权限等级划分
| 权限等级 | 允许执行的操作 | 平台要求 |
|----------|----------------|----------|
| 普通用户权限 | 1. 所有只读检查、诊断、监控操作<br>2. 当前用户进程的端口查询、进程查看<br>3. 当前用户的启动项查看<br>4. 当前用户目录的磁盘扫描、文件分析<br>5. 所有报告、快照导出操作 | 无特殊要求，三平台通用 |
| 特权权限（root/管理员） | 1. 全系统所有进程、端口、服务、启动项的查看<br>2. 结束系统进程、修改系统服务、禁用系统启动项<br>3. 系统级目录的磁盘扫描、文件清理<br>4. 系统级配置修改、定时任务配置 | Linux/macOS：sudo/root权限<br>Windows：以管理员身份运行 |

### 6.2 权限降级策略
1.  普通用户执行需要特权权限的操作时，不会报错退出，而是自动降级为普通用户权限可执行的范围；
2.  输出结果中标注**覆盖率**，计算公式：`覆盖率 = 已成功采集的模块数 / 总模块数 * 100%`；
3.  对无权限访问的内容，标注`权限不足，无法访问`，不中断整体命令执行；
4.  终端输出明确提示：`当前为普通用户权限，部分数据无法采集，建议使用root/管理员权限执行以获得完整结果`。

### 6.3 安全执行机制
1.  **默认只读原则**：所有修改系统的写操作，默认开启dry-run模式，仅模拟执行，不实际修改系统，必须显式指定--apply才会执行；
2.  **危险操作二次确认**：dangerous级别的操作，必须同时指定--apply和--yes才能执行，否则会弹出二次确认提示，用户手动确认后才会执行；
3.  **操作审计日志**：所有写操作、危险操作，都会写入审计日志，记录：执行时间、执行用户、命令、参数、执行结果、IP地址、进程PID；
4.  **备份回滚机制**：修改系统配置、禁用启动项、删除文件等操作，执行前自动备份原配置/文件，支持一键回滚；
5.  **白名单机制**：系统关键进程、核心服务默认加入白名单，禁止结束/修改，除非用户显式指定--force参数强制操作。

### 6.4 错误处理与降级策略
1.  单个模块采集/执行失败时，不影响其他模块的执行，最终输出汇总失败模块清单和错误原因；
2.  模块失败时，健康分计算会扣除对应模块的权重，标注`未检查`，不会因为单个模块失败导致整个体检失败；
3.  平台不支持的功能，自动跳过，标注`当前平台不支持该功能`，不报错退出；
4.  所有异常都有明确的错误提示、错误码、修复建议，不会出现无意义的panic。

### 6.5 错误提示规范

所有错误提示必须遵循统一的格式，包含错误码、错误描述、建议操作，方便用户快速定位和解决问题。

#### 6.5.1 错误码定义

| 错误码 | 错误类型 | 适用场景 | 退出码 |
|--------|----------|----------|--------|
| ERR_INVALID_ARGUMENT | 参数错误 | 命令行参数非法、缺失、格式错误 | 3 |
| ERR_PERMISSION_DENIED | 权限不足 | 当前用户无权限执行操作 | 4 |
| ERR_PLATFORM_UNSUPPORTED | 平台不支持 | 当前平台不支持该功能 | 5 |
| ERR_EXECUTION_FAILED | 执行失败 | 系统命令执行失败、外部依赖异常 | 5 |
| ERR_CONFIG_INVALID | 配置错误 | 配置文件格式错误、字段非法 | 3 |
| ERR_TIMEOUT | 超时 | 模块执行超时 | 5 |
| ERR_NOT_FOUND | 资源不存在 | 文件、进程、端口等资源不存在 | 5 |
| ERR_ALREADY_EXISTS | 资源已存在 | 快照、配置文件等已存在 | 5 |
| ERR_STORAGE_FULL | 存储空间不足 | 磁盘空间不足、存储配额超限 | 5 |

## 7. 跨平台适配规范
### 7.1 平台适配范围
| 平台 | 最低支持版本 | 适配说明 |
|------|--------------|----------|
| Linux | 内核3.10+ | 优先适配systemd发行版（CentOS 7+/Ubuntu 16.04+/Debian 9+），兼容sysvinit发行版 |
| Windows | Windows 10+/Windows Server 2016+ | 兼容x86/x64/arm64架构，适配WMI、Windows服务、注册表 |
| macOS | macOS 10.15+ | 兼容Intel/Apple Silicon芯片，适配launchd、sysctl、系统日志 |

### 7.2 适配原则
1.  **统一接口原则**：所有平台相关的采集、执行操作，都在platform层封装统一的接口，上层模块无需关心平台差异；
2.  **功能一致性原则**：三平台的核心命令、参数、输出格式完全一致，平台专属功能会明确标注；
3.  **路径适配原则**：自动适配各平台的路径分隔符、系统目录、用户目录，用户无需手动适配；
4.  **无cgo原则**：所有适配均采用纯Go实现，不依赖cgo，保证静态编译、跨平台交叉构建的便捷性，100%复用原有构建脚本。

### 7.3 跨平台依赖的系统命令清单

各模块在不同平台上依赖的系统命令和API，开发时需要做好适配：

#### 7.3.1 端口管理模块（port）
| 功能 | Linux | Windows | macOS | 备注 |
|------|-------|---------|-------|------|
| 端口占用查询 | `ss -tunlp` 或 `netstat -tunlp` | `netstat -ano` 或 WMI | `lsof -i` 或 `netstat -anv` | 优先使用 gopsutil.net.Connections() |
| 进程与端口映射 | `/proc/net/tcp` + `/proc/<pid>/` | WMI Win32_Process | `lsof -i -P` | gopsutil 已封装 |
| 端口扫描 | 原生 TCP 连接 | 原生 TCP 连接 | 原生 TCP 连接 | 使用 Go net 包 |

#### 7.3.2 进程管理模块（proc）
| 功能 | Linux | Windows | macOS | 备注 |
|------|-------|---------|-------|------|
| 进程列表 | `/proc/` 目录 | WMI Win32_Process | `ps` 或 sysctl | 使用 gopsutil.process |
| 进程树 | `/proc/<pid>/status` | WMI ParentProcessId | `ps -o ppid` | gopsutil 已封装 |
| 进程资源 | `/proc/<pid>/stat` | WMI WorkingSetSize/UserModeTime | `ps -o %cpu,%mem` | gopsutil.process.Percent() |
| 结束进程 | `kill -9 <pid>` | `taskkill /F /PID <pid>` | `kill -9 <pid>` | 使用 process.Kill() |

#### 7.3.3 CPU诊断模块（cpu）
| 功能 | Linux | Windows | macOS | 备注 |
|------|-------|---------|-------|------|
| CPU使用率 | `/proc/stat` | WMI Win32_Processor | `sysctl kern.cp_time` | 使用 gopsutil.cpu.Percent() |
| CPU核心数 | `/proc/cpuinfo` | WMI NumberOfLogicalProcessors | `sysctl hw.ncpu` | gopsutil.cpu.Counts() |
| 进程CPU占用 | `/proc/<pid>/stat` | WMI PercentProcessorTime | `ps -p <pid> -o %cpu` | gopsutil.process.Percent() |

#### 7.3.4 内存诊断模块（mem）
| 功能 | Linux | Windows | macOS | 备注 |
|------|-------|---------|-------|------|
| 内存总量/使用 | `/proc/meminfo` | WMI Win32_OperatingSystem | `sysctl hw.memsize` | 使用 gopsutil.mem.VirtualMemory() |
| 进程内存占用 | `/proc/<pid>/status` | WMI WorkingSetSize | `ps -p <pid> -o rss` | gopsutil.process.MemoryInfo() |
| Swap使用 | `/proc/swaps` | WMI Win32_PageFileUsage | `sysctl vm.swapusage` | gopsutil.mem.SwapMemory() |

#### 7.3.5 磁盘管理模块（disk）
| 功能 | Linux | Windows | macOS | 备注 |
|------|-------|---------|-------|------|
| 磁盘分区列表 | `df -h` 或 `/proc/mounts` | WMI Win32_LogicalDisk | `df -h` | 使用 gopsutil.disk.Partitions() |
| 磁盘使用率 | `statvfs()` | GetDiskFreeSpaceEx | `statvfs()` | gopsutil.disk.Usage() |
| 文件扫描 | `filepath.Walk()` | `filepath.Walk()` | `filepath.Walk()` | Go 标准库 |
| 文件哈希 | `crypto/sha256` | `crypto/sha256` | `crypto/sha256` | Go 标准库 |

#### 7.3.6 网络管理模块（net）
| 功能 | Linux | Windows | macOS | 备注 |
|------|-------|---------|-------|------|
| 网络连接列表 | `ss -tunap` 或 `/proc/net/tcp` | `netstat -ano` | `netstat -an` | 使用 gopsutil.net.Connections() |
| DNS解析 | `nslookup` 或 `dig` | `nslookup` | `nslookup` 或 `dig` | 使用 Go net.LookupHost() |
| Ping测试 | `ping` 或 ICMP socket | `ping` 或 ICMP socket | `ping` 或 ICMP socket | 使用第三方库或原生 ICMP |
| 路由跟踪 | `traceroute` | `tracert` | `traceroute` | 调用系统命令 |
| 网络接口信息 | `/sys/class/net/` | WMI Win32_NetworkAdapter | `ifconfig` | gopsutil.net.Interfaces() |

#### 7.3.7 服务管理模块（service）
| 功能 | Linux | Windows | macOS | 备注 |
|------|-------|---------|-------|------|
| 服务列表 | `systemctl list-units` 或 `service --status-all` | `sc query` 或 WMI Win32_Service | `launchctl list` | 需要平台分支实现 |
| 服务状态 | `systemctl status <service>` | `sc query <service>` | `launchctl list <service>` | 需要平台分支实现 |
| 启动/停止服务 | `systemctl start/stop <service>` | `sc start/stop <service>` | `launchctl start/stop <service>` | 需要管理员权限 |

#### 7.3.8 启动项管理模块（startup）
| 功能 | Linux | Windows | macOS | 备注 |
|------|-------|---------|-------|------|
| 启动项列表 | `/etc/systemd/system/` 或 `/etc/init.d/` | 注册表 `HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run` | `~/Library/LaunchAgents/` | 需要平台分支实现 |
| 禁用启动项 | `systemctl disable <service>` | 修改注册表 | `launchctl unload <plist>` | 需要管理员权限 |

#### 7.3.9 日志分析模块（log）
| 功能 | Linux | Windows | macOS | 备注 |
|------|-------|---------|-------|------|
| 系统日志路径 | `/var/log/` | `C:\Windows\System32\winevt\Logs\` | `/var/log/` | 需要平台适配 |
| 日志读取 | 直接读取文件 | Windows Event Log API | 直接读取文件或 `log show` | 需要平台分支实现 |

**适配策略**：
1. **优先使用 gopsutil**：对于 CPU、内存、磁盘、进程、网络等基础指标，优先使用 gopsutil 库，它已经封装了跨平台差异
2. **Go 标准库次之**：文件操作、网络连接、哈希计算等使用 Go 标准库
3. **系统命令兜底**：对于 gopsutil 不支持的功能（如服务管理、启动项管理），使用 `os/exec` 调用系统命令
4. **平台分支隔离**：将平台相关代码隔离到 `platform/windows/`、`platform/linux/`、`platform/darwin/` 目录

2.  所有只读查询类命令，执行耗时≤2s；
3.  大目录磁盘扫描，速度不低于当前仓库迁移前的扫描能力；
4.  持续监控模式，CPU占用≤1%，内存占用≤50MB；
5.  重复文件扫描，支持100万+文件的目录扫描，无内存溢出。

4.  普通用户和root/管理员权限下，均能正常执行，降级策略正常。

### 10.4 安全验收标准

2.  dangerous级操作无--yes参数时，不会执行，必须二次确认；

### 11.3 配置文件加载逻辑

配置文件加载流程和优先级处理：

```go
// internal/config/loader.go
package config

import (
    "os"
    "path/filepath"
    "runtime"

    "gopkg.in/yaml.v3"
)

// LoadConfig 加载配置文件
// 优先级：命令行参数 > 用户配置 > 系统配置 > 默认值
func LoadConfig() (*Config, error) {
    // 1. 加载默认配置
    cfg := DefaultConfig()

    // 2. 加载系统级配置（如果存在）
    systemConfigPath := GetSystemConfigPath()
    if fileExists(systemConfigPath) {
        if err := mergeConfigFromFile(cfg, systemConfigPath); err != nil {
            return nil, err
        }
    }

    // 3. 加载用户级配置（如果存在）
    userConfigPath := GetUserConfigPath()
    if fileExists(userConfigPath) {
        if err := mergeConfigFromFile(cfg, userConfigPath); err != nil {
            return nil, err
        }
    }

    // 4. 命令行参数会在命令执行时覆盖配置

    return cfg, nil
}

// DefaultConfig 返回默认配置
func DefaultConfig() *Config {
    return &Config{

// GetSystemConfigPath 获取系统级配置路径
func GetSystemConfigPath() string {
    switch runtime.GOOS {
    case "windows":
        return filepath.Join(os.Getenv("ProgramData"), "syskit", "config.yaml")
    case "darwin":
        return "/Library/Application Support/syskit/config.yaml"
    default: // linux
        return "/etc/syskit/config.yaml"
    }
}

// GetUserConfigPath 获取用户级配置路径
func GetUserConfigPath() string {
    homeDir, _ := os.UserHomeDir()
    return filepath.Join(homeDir, ".syskit", "config.yaml")
}

// mergeConfigFromFile 从文件合并配置
func mergeConfigFromFile(cfg *Config, path string) error {
    data, err := os.ReadFile(path)
    if err != nil {
        return err
    }

    // 解析 YAML 并合并到现有配置
    return yaml.Unmarshal(data, cfg)
}

// fileExists 检查文件是否存在
func fileExists(path string) bool {
    _, err := os.Stat(path)
    return err == nil
}

func getDefaultLogPath() string {
    homeDir, _ := os.UserHomeDir()
    return filepath.Join(homeDir, ".local", "share", "syskit", "logs", "syskit.log")
}

func getDefaultDataDir() string {
    homeDir, _ := os.UserHomeDir()
    return filepath.Join(homeDir, ".local", "share", "syskit", "data")
}
```

### 11.4 审计日志格式定义

审计日志用于记录所有写操作，格式为 JSONL（每行一个 JSON 对象）：

**日志文件路径**：
- Linux/macOS: `~/.local/share/syskit/data/audit/audit_YYYY-MM-DD.jsonl`
- Windows: `%LOCALAPPDATA%/syskit/data/audit/audit_YYYY-MM-DD.jsonl`

**日志字段定义**：
```go
// internal/audit/audit.go
package audit

import (
    "encoding/json"
    "os"
    "path/filepath"
    "time"
)

// AuditLog 审计日志结构
type AuditLog struct {
    Timestamp  time.Time              `json:"timestamp"`
    TraceID    string                 `json:"trace_id"`
    Operator   string                 `json:"operator"`
    Command    string                 `json:"command"`
    Action     string                 `json:"action"`
    Target     string                 `json:"target"`
    Before     interface{}            `json:"before,omitempty"`
    After      interface{}            `json:"after,omitempty"`
    Result     string                 `json:"result"`
    ErrorMsg   string                 `json:"error_msg,omitempty"`
    DurationMs int64                  `json:"duration_ms"`
    Metadata   map[string]interface{} `json:"metadata,omitempty"`
}

// Logger 审计日志记录器
type Logger struct {
    logDir string
}

// NewLogger 创建审计日志记录器
func NewLogger(dataDir string) *Logger {
    return &Logger{
        logDir: filepath.Join(dataDir, "audit"),
    }
}

// Log 记录审计日志
func (l *Logger) Log(log *AuditLog) error {
    // 确保日志目录存在
    if err := os.MkdirAll(l.logDir, 0755); err != nil {
        return err
    }

    // 按日期分文件
    filename := time.Now().Format("audit_2006-01-02.jsonl")
    filepath := filepath.Join(l.logDir, filename)

    // 打开文件（追加模式）
    file, err := os.OpenFile(filepath, os.O_CREATE|os.O_WRONLY|os.O_APPEND, 0644)
    if err != nil {
        return err
    }
    defer file.Close()

    // 序列化为 JSON
    data, err := json.Marshal(log)
    if err != nil {
        return err
    }

    // 写入文件（每行一个 JSON）
    _, err = file.Write(append(data, '\n'))
    return err
}
```

### 11.5 常见问题（FAQ）

**Q1: 如何调试跨平台问题？**
A:
1. 使用 `runtime.GOOS` 判断当前平台
2. 在 `platform/` 目录下为每个平台实现专属逻辑
3. 使用 Docker 或虚拟机在本地测试其他平台
4. 在 CI/CD 中配置三平台自动化测试

**Q2: 如何处理权限不足的情况？**
A:
1. 使用 `os.Getuid()` 检查当前用户权限（Linux/macOS）
2. Windows 使用 `golang.org/x/sys/windows` 检查管理员权限
3. 权限不足时，降级执行，只采集当前用户可访问的数据
4. 在输出中明确标注权限不足的模块

**Q3: 如何测试 dry-run 模式？**
A:
1. 所有写操作函数接收一个 `dryRun bool` 参数
2. dry-run 模式下，只输出将要执行的操作，不实际执行
3. 单元测试中同时测试 dry-run 和实际执行两种模式
4. 集成测试中验证 dry-run 不会修改系统状态

**Q4: 配置文件不存在时怎么办？**
A:
配置文件不存在是正常情况，使用默认配置即可。用户可以通过 `syskit policy init` 生成配置文件模板。

**Q5: 配置文件格式错误时怎么办？**
A:
1. YAML 解析失败时，返回 `ERR_CONFIG_INVALID` 错误
2. 错误信息中包含具体的错误行号和字段名
3. 建议用户使用 `syskit policy validate` 验证配置文件

**Q6: 如何合并多个配置文件？**
A:
使用 `yaml.Unmarshal` 多次解析到同一个结构体，后面的配置会覆盖前面的配置。优先级：命令行 > 用户配置 > 系统配置 > 默认值。

**Q7: 如何实现跨平台的进程结束？**
A:
使用 `gopsutil` 库的 `process.Kill()` 方法，它已经封装了跨平台差异。

**Q8: 如何处理大文件扫描的内存问题？**
A:
1. 使用 `filepath.Walk` 流式遍历，不要一次性加载所有文件
2. 使用堆（heap）维护 TopN 结果，避免存储所有文件信息
3. 对于重复文件检测，先按大小分组，只对同大小的文件计算哈希

**Q9: 如何实现命令行参数覆盖配置文件？**
A:
1. 先加载配置文件
2. 在 cobra 命令的 `PreRun` 或 `Run` 中，检查命令行参数是否设置
3. 如果命令行参数非默认值，覆盖配置文件中的值

**Q10: 如何实现三平台一致的路径处理？**
A:
1. 始终使用 `filepath.Join` 拼接路径，不要手动拼接
2. 使用 `filepath.ToSlash` 和 `filepath.FromSlash` 转换路径分隔符
3. 配置文件中的路径使用 `/`，加载时自动转换为平台路径

## 11. 风险与规避措施
| 风险类型 | 风险描述 | 规避措施 |
|----------|----------|----------|
| 跨平台适配风险 | 三平台的系统接口、服务管理、启动项路径差异大，适配难度高 | 1. 采用gopsutil成熟库屏蔽底层差异<br>2. 平台专属逻辑全部封装在platform层，上层无平台相关代码<br>3. 每个版本都做三平台全量测试 |
| 性能风险 | 全量扫描、大目录分析、高频监控可能导致高资源占用 | 1. 采用并发控制，限制最大并发数<br>2. 大文件扫描采用流式处理，避免内存溢出<br>3. 高频采样采用定时器优化，降低CPU占用 |
| 安全风险 | 修复操作、进程结束、文件删除可能导致用户误操作，影响系统稳定性 | 1. 严格执行默认dry-run、危险操作二次确认机制<br>2. 系统关键操作加入白名单保护<br>3. 所有修改操作执行前自动备份，支持回滚 |
| 范围蔓延风险 | 功能迭代过程中不断新增需求，导致MVP版本无法按时交付 | 1. 严格按照P0/P1/P2优先级迭代，MVP版本仅交付P0功能<br>2. 新增需求先纳入需求池，下个版本迭代评估<br>3. 严格遵循YAGNI原则，不做非必要功能 |
